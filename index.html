记录空指针踩过的坑

最近因为空指针踩过了一个又一个的坑，今天打算好好总结一下

首先因为空指针是在进行数据库操作的时候没有实例化dao层的impl，导致无法进行数据库操作，那么具体的场景如下：

针对数据库的操作首先定义了dao层的接口，该层指定了抽象的增删改查的方法，但是没有定义具体的实现。具体的实现在impl层定义的。然后使用的时候引入了dao层的依赖包

在执行update的时候，项目报了空指针，排查bug发现没啥问题，为啥会报空指针呢，后来在同事的协助之下发现是因为没有引入impl包，导致没有对dao层提供具体的实现，即实例化这个dao。那么解决方式就是再引入impl层的依赖包。

这样就解决了空指针的问题。

乐观锁的实战

最近在做取消支付的时候用到了锁操作来保证并发性，一方面通过悲观锁进行一个全局的流程的加锁，然后在进行数据库表的更新时，在进行一个乐观锁机制来保障并发安全。

全局的流程的加锁：

在进行数据库表更新时，采用乐观锁的机制来进一步保证并发安全：

具体的做法是在订单表里面添加一个字段：casVersion 用来实现乐观锁，即更新的时候再去检测数据是否冲突，如果冲突就报错

在更新数据库表的时候，先检验casVersion是否等于预期值，相等则更新为newVersion，这个逻辑也和CAS(compare and swap)的原理一致，相等则交换。

CAS执行过程是这样：它包含 3 个参数 CAS（V，E，N），V表示要更新变量的值，E表示预期值，N表示新值。仅当 V值等于E值时，才会将V的值设为N，如果V值和E值不同，则说明已经有其他线程完成更新，则当前线程则什么都不做，最后CAS 返回当前V的真实值。

点开网址：https://developers.google.com/protocol-buffers/docs/reference/go/faq#namespace-conflict

可以看到有提供以下的解决方式：

  1. At compile time. The default behavior for handling conflicts can be specified at compile time with a linker-initialized variable: go build -ldflags "-X google.golang.org/protobuf/reflect/protoregistry.conflictPolicy=warn"
  2. At program execution. The behavior for handling conflicts when executing a particular Go binary can be set with an environment variable: GOLANG_PROTOBUF_REGISTRATION_CONFLICT=warn ./main

设置环境变量可以通过在编辑器的edit config方式设置：

GOLANG_PROTOBUF_REGISTRATION_CONFLICT=warn

总结：有的时候报错的内容里面其实就有给具体的解决方式，需要认真地去查看报错内容
