旁路缓存模式

3.2.1 category blacklist的缓存一致性

针对3.2.0中涉及的API的特点，对category blacklist的缓存一致性进行设计。针对category blacklist的缓存场景的特点可以总结如下：

  • 读多写少(写的频率较低)：category blacklist主要用于在PDP的详情页访问时进行商品类别的黑名单过滤，因此该场景需要扛住大量的访问请求，但是对黑名单的更新主要在admin侧，且更新的频率较低？
  • 最终一致性：可以容忍在一定的时延内数据出现不一致，此时PDP的详情页命中黑名单的商品仍然展示SPL的分期信息，但最终需达到最终一致性的要求
  • 数据量不是很大：黑名单的数据并不想订单等数据量庞大
1. 方案一：旁路缓存模式(Cache Aside Pattern)
  • 核心思想：适用于读多写少的场景，服务端同时维系DB和cache，并且以DB的结果为准。

​ 写：Admin侧更新DB中的category之后，然后清除Redis中的category blacklist缓存（清除所有？）

​ 读：Pricing Center侧进行category过滤时，首先读取Redis, 若cache命中返回数据；

​ 若cache未命中，则访问DB，若读取到数据返回数据，然后放入到cache中，若未读取到数据直接返回。

  • 该方案的缺陷：

    1）首次请求数据一定不在cache中。

    2）写操作比较频繁的话导致cache中的数据会被频繁被删除，从而影响缓存命中率。【这里category blacklist的更新操作主要有后台ops操作，更新频率应该不高？】

  • 是否适用category blacklist的缓存

category blacklist的缓存场景为读多写少的场景，因此可采用旁路缓存模式。但由于查询产品分期query_product_instalment_info访问量较大，对QPS的要求较高，因此当cache未命中时，不能直接访问DB。

实现方式：

  • 通过定时任务来定期的刷新缓存来实现最终一致性，【将过期时间设置较长一点，因为更新频率较低，定时任务的周期也不用过于频繁，从而提高命中率】

  • 另外可考虑在数据库中给category设置过期时间?（与Redis的过期时间如何兼容，数据库中过期主动更新？）

另外针对该缓存模式的缺陷可采用以下方式解决：

1）在大促时，针对热点数据需要考虑提前预热，提前将其放入redis缓存中,以防止大量请求同时访问DB，导致缓存雪崩。

2）防止在大促期间进行黑名单的更新，降低对缓存命中率的影响。

3）在该场景下，可以短暂地允许数据库和缓存数据不一致，因此可以给category list加一个较短的过期时间，降低数据不一致的影响（好像与前面有点矛盾，权衡数据的不一致性和命中率）。

2. 方案二：读写穿透模式(Read/Write Though Pattern)
  • 核心思想：服务端将cache视为主要数据存储，直接读取和接入Redis，并由cache服务负责将数据读取和写入DB

写：先查cache，cache中不存在，直接更新DB；若cache中存在，先更新cache，再调用cache服务同步更新DB

读：先查cache，命中则直接返回；否则cache服务会先从DB加载数据然后自己写入redis缓存中，再返回响应。

  • 缺陷：

1）首次请求的数据一定不在cache中。

  • 是否适用category blacklist的存储

​ 该模式显然不适用于category blacklist的缓存案例，因为这里的写只能是Admin侧写入DB，然后Pricing center侧负责从redis中读取数据，不能通过Redis向DB中写入数据。

\3. 方案三：异步缓存写入(Write Behind Pattern)

  • 核心思想：与读写穿透模式相似，由cache服务负责cache和DB的读写。适用于频繁写的场景，但对数据的一致性要求不强的场景

写：先查cache，cache未命中，直接更新DB；若cache命中，先更新cache，然后调用cache服务异步更新DB

读：先查cache，命中直接返回响应；若未命中，cache服务会先从DB加载数据，然后自己写入Cache，然后返回响应。

  • 是否适用category blacklist的存储

该缓存模式同样不适用于category blacklist的存储，因为category blacklist是读请求较多，写请求较少的场景，且对数据的一致性要求较高。

消费订阅模式：

使用 Binlog 实时更新 Redis 缓存

这种收 Binlog 更新缓存的方案，和刚刚我们讲到的，收 MQ 消息更新缓存的方案，其实它们的实现思路是一样的，都是异步订阅实时数据变更信息，去更新 Redis。只不过，直接读取 Binlog 这种方式，它的通用性更强。不要求订单服务再发订单消息了，订单更新服务也不用费劲去解决「发消息失败怎么办？」这种数据一致性问题了。

而且，在整个缓存更新链路上，减少了一个收发 MQ 的环节，从 MySQL 更新到 Redis 更新的时延更短，出现故障的可能性也更低，所以很多大厂更青睐于这种方案。

这个方案唯一的缺点是，实现订单缓存更新服务有点儿复杂，毕竟不像收消息，拿到的直接就是订单数据，解析 Binlog 还是挺麻烦的。
